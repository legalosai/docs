<Update label="July 13, 2026" description="Week of July 13">

## New features

**E-3 petitions on platform.** You can now run E-3 (Australian specialty occupation) petitions end-to-end in LegalOS, alongside H-1B and other supported visa types.

- Full E-3 intake for both employer and beneficiary, including a dedicated current immigration status step, specialty occupation details, and processing options.
- Australian-nationality validation on the beneficiary side to catch eligibility issues before submission.
- Auto-generation of the E-3 form set (I-129, LCA, G-28, I-907) with E-3–specific I-129 mapping: classification set to E-3, basis 2.b, requested action 4.c, and Part 3 populated from the beneficiary's current status.
- Filings are routed to the Vermont Service Center (VSC) by default.
- No evidence module is required for E-3, so the workflow skips it automatically.

## Updates

- **Shared address component for H-1B secondary worksites.** The H-1B secondary work-location address now uses the same standardized address input as the rest of the platform, for a more consistent experience and cleaner data.

## Bug fixes

- **H-1B second worksite toggle now persists.** Unchecking "second physical work location" on an H-1B petition now correctly removes the secondary worksite instead of leaving stale data behind.

</Update>

<Update label="July 10, 2026" description="Week of July 6 – July 10">

## New features

**USCIS filing-fee Stripe invoicing.** Staff can now build itemized filing-fee invoices for a petition directly from the admin sidebar and send the client a single Stripe invoice with a hosted invoice page and pay link.

- Build a breakdown from the seeded USCIS filing-fee catalog via a dropdown, plus add editable custom line items.
- All amounts are staff-editable before sending, so you can reflect the current fee schedule or one-off adjustments per petition.
- The recipient signatory is resolved using the existing employer-signatory logic, and the Stripe customer's email stays in sync with the selected recipient.
- Sent (unpaid) invoices can be voided from the admin — the invoice is voided in Stripe and the local record moves to a `void` state.
- Send, finalize, and void calls are idempotent, so a double-click or retry can't create duplicate customers or invoices.

Filing-fee invoicing is gated on a Stripe API key. Until it's configured, the card shows a graceful "not configured" state and no invoicing actions run.

## Updates

- **Filing group in the admin sidebar.** The Filing Fee Invoice card lives in the sidebar's Filing group, and sidebar navigation now scrolls the target section into view below the sticky top nav.

## Coming soon

- Paid-status webhook and a client-facing invoice to-do so applicants can see and pay their filing-fee invoice from their portal.
